## Durable Reactor API Direction For AI Workflows

The public API should stay centered on normal Reactor `step` declarations plus
durability `mode`, not on a separate `agent_step` abstraction.

That means:

- keep `step`
- keep `input`
- keep `argument`
- keep `result`
- keep `mode :replayable | :resumable`
- do not introduce a separate `needs do` block
- do not make `await_resume` the conceptual center; it is only sugar over a resumable step

The important rule is:

> the step's declared dependency set is the persisted execution payload boundary

If a step declares:

- `argument :config, input(:config)`
- `argument :week_data, result(:read_weekly_checkin)`
- `argument :delivery, result(:send_message)`

then the runtime should persist the fully resolved payload for that step by default.

That payload should include:

- the referenced run inputs
- the resolved step arguments
- the referenced upstream step results

No separate `persist_context`-style selection should be required for the AI workflow case.
The step already declares its dependency surface through normal Reactor wiring.

### Example Desired DSL

```elixir
defmodule MyApp.AccountabilityWorkflow do
  use Reactor, extensions: [AshDurableReactor]

  durable do
    sqlite repo: MyApp.Repo
  end

  input :config
  input :phase

  step :extract_inputs do
    mode :replayable
    argument :config, input(:config)
    run MyApp.Steps.ExtractInputs
  end

  step :read_weekly_checkin do
    mode :replayable
    argument :config, input(:config)
    argument :inputs, result(:extract_inputs)
    run MyApp.Steps.ReadWeeklyCheckin
  end

  step :build_message do
    mode :replayable
    argument :config, input(:config)
    argument :phase, input(:phase)
    argument :inputs, result(:extract_inputs)
    argument :week_data, result(:read_weekly_checkin)
    run MyApp.Steps.BuildMessage
  end

  step :checkin_reply do
    mode :resumable
    argument :message, result(:build_message)
    run MyApp.Steps.WaitForReply
  end

  step :record_checkin do
    mode :replayable
    argument :config, input(:config)
    argument :phase, input(:phase)
    argument :inputs, result(:extract_inputs)
    argument :week_data, result(:read_weekly_checkin)
    argument :message, result(:build_message)
    argument :reply, result(:checkin_reply)
    run MyApp.Steps.RecordCheckin
  end
end
```

### What The Runtime Should Persist

For a step like `:record_checkin`, the persisted step payload should contain the
resolved data that actually formed the execution boundary:

```elixir
%{
  inputs: %{
    config: %{...},
    phase: :run
  },
  arguments: %{
    inputs: %{...},
    week_data: %{...},
    message: %{...},
    reply: %{...}
  }
}
```

This is better than one giant workflow blob as the default.

It means:

- each step gets exactly the data it declared
- replay stays local and understandable
- resumable steps can restart with the same resolved payload
- agent workflows remain simple because rich dependency payloads are available by default

### Delegated Type Interpretation

Rails delegated type should influence the storage model more than the DSL.

The equivalent shape here is:

- `durable_runs` = stable shared wrapper for a run
- `durable_steps` = stable shared wrapper for each step execution
- `durable_payloads` = typed payload records for resolved inputs, outputs, resume payloads, errors

That is:

- wrapper records change state
- payload records hold the specific execution data
- event/history records track transitions over time

So the long-term direction is not "serialize the entire workflow state into one map".
The better direction is:

- persist step-scoped resolved payloads
- persist step outputs separately
- link them from stable run/step wrapper records

### Important API Conclusions

1. Do not add `agent_step` as a core abstraction.
2. Do not add a separate `needs do` block.
3. Normal Reactor `input` / `argument` / `result` declarations already define what the step needs.
4. `mode` should be the central durability primitive.
5. `await_resume` can remain as convenience sugar, but resumable behavior must fundamentally belong to normal steps.
6. Avoid `current_step_index`; it is not a useful public durability concept.
7. `persist_context` is too narrow for agent workflows and should not be the main model for this use case.

### Immediate Design Direction

The next meaningful iteration of this library for AI workflows should:

- persist the fully resolved execution payload for each step
- resume resumable steps with that same resolved payload plus the resume payload
- keep run-level metadata light
- use delegated-type thinking for storage and event history, not for complicating the step DSL

## Original Notes From Chat On Delegated Type

By separating the mechanics of existence and execution from the specific
details of the task, we can build an architecture that scales without becoming
 tangled.

The core philosophy of the delegated type pattern is separating shared
metadata/behavior from specific underlying data.

### Recording

A single table that holds everything common across diverse objects.

This includes:

- timestamps
- authorship
- global IDs
- routing URLs
- access control / buckets

This layer is mutable, meaning state changes happen here.

### Recordable

The actual implementation data.

Examples:

- a comment's text
- a document's markdown
- an agent's specific prompt

These are highly specific, decoupled tables, and are generally immutable or
append-only.

### Parent Child And Hierarchy Unity

Because every entity is fundamentally a Recording, building unified trees is
trivial.

Examples:

- a Document (Recording) can have Comments (Recordings)
- one query can fetch a chronological timeline of all activity in a Bucket

### Composable Behaviors

Implement a behavior once on the Recording.

Example:

- a Recording has an `is_commentable` boolean
- by default it is false
- a Document implementation overrides it to true
- if later Comments also become commentable, flip the boolean and the
  infrastructure is already there

### What To Steal For Agents

The key parts worth stealing for the agent system are:

- a stable shared record that rarely changes shape
- immutable subtype payloads
- an event/history layer that records which payload version the shared record
  pointed to at a given time

This is the right mental model for the durable agent workflow system too.
